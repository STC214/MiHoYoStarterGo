# MiHoYoStarterGo

基于 `Go + Wails + Vue3` 的米哈游账号切换与登录辅助工具（本地离线 OCR）。
显著缺点是ocr方案识别速度较慢，（根据pc配置程度通常需要5-20s）。而且有时候会莫名其妙的识别不到（就很烦，不过动态画面时常会有类似的问题）。

当然相较于缺点也有一些让人舒服的优点：
1.ocr识别可以应对的场景和操作更灵活
2.每次登陆都会记录当前账号在系统里的存档，显著降低账号过期需要重新登陆的问题
3.如果确认登陆成功可以自己点击手动确认登陆成功来缩短整体流程。
4.纯本地应用，不联网。

## 当前状态
- 支持游戏：
  - `GenshinCN`（原神国服）
  - `StarRailCN`（崩坏：星穹铁道国服）
  - `ZZZCN`（绝区零国服）
- 程序可最小化到托盘，支持托盘菜单恢复与双击图标恢复。

## 核心能力
- 本地账号管理（增删改、排序、密码加密存储）
- 游戏路径管理
- 登录流程状态控制（启动、暂停、停止监控）
- OCR 驱动的界面识别与自动登录
- 登录成功后读取注册表 Token 与硬件信息并写回配置

## 登录流程（最新）
- 启动画面预热阶段（新增）：
  - 游戏进程刚启动时，常见“白底黑字”的游戏标题/忠告画面会被视为预热阶段
  - 该阶段**不进行 OCR 识别**，直接跳过
  - 仅当画面属于“纯黑白/白黑”启动页时跳过 OCR；离开该画面后立即开始 OCR
- 识别分组（并行检测）：
  - 原神：`成功组（点击进入/开始游戏）`、`登录同意组（登录框/同意）`
  - 崩铁：`成功组（点击进入/开始游戏/登出）`、`登录同意组`、`+86/账号密码组`
  - 绝区零：`成功组（点击进入）`、`登录组（坐标流程）`、`+86/账号密码组`
  - 每帧会同时检测所有组；任一成功组命中即结束
  - 同一帧动作执行采用互斥优先级，避免多组动作互相打断
- OCR 轮询间隔：
  - 原神（`GenshinCN/GenshinOS`）轮询间隔固定为 `300ms`
- 账号密码阶段：
  - 识别并填写账号、密码
  - 输入完成后只点击一次 `进入游戏`
- 协议弹窗阶段：
  - 只有在**同一帧 OCR 同时识别到** `不同意` 和 `同意` 时，才允许执行同意点击
  - 点击策略：优先点击“不同意”后紧跟的“同意”；若未命中，回退到同排右侧最近“同意”；再不行则兜底点击最后一个纯“同意”
  - 点击后等待约 1 秒再继续识别
- 成功判定阶段：
  - 识别到 `点击进入` 或 `开始游戏` 视为登录成功
  - 崩铁额外特征：`点击进入`（需位于画面底部 1/5）或 `登出`（需位于画面右侧 1/10）也会判定为成功
  - 崩铁成功判定包含 OCR 文本识别 + 图片模糊匹配双通道并行；任一命中后会取消另一通道并完成回收，减少半透明 UI 导致的漏判
  - 成功后保存注册表 Token 与硬件信息，流程结束

## 绝区零模块（当前实现）
- 采用坐标流程（不与 OCR 登录填写流程混跑）。
- 画面 B 支持手动标定以下坐标：
  - 账号输入框
  - 密码输入框
  - 同意协议勾选点
  - 进入游戏点击点
- 坐标按窗口相对位置保存，并按分辨率进行匹配。

## 安全与数据
- 所有识别与自动化流程均在本机执行，不依赖云端 OCR。
- 配置文件默认位于程序可执行文件（`exe`）同目录：`config.json`
- 程序会优先读取 `exe` 同目录配置；若该处不存在，再回退读取当前工作目录配置（兼容开发态）
- 通过外部启动器（含管理员权限启动）拉起时，即使工作目录变化，也会优先使用自身 `exe` 目录下的配置文件

## 项目结构（主要）
- 后端入口：`app.go`
- 业务层：`app_logic/`
- 核心逻辑：`logic/`
- 前端：`frontend/`

## 本地运行
1. 安装依赖（Go、Node、Wails）
2. 安装前端依赖：
   - `cd frontend`
   - `npm install`
3. 回到项目根目录运行：
   - `wails dev`

## 构建
- 前端构建：
  - `cd frontend`
  - `npm run build`
- Go 编译检查（项目根目录）：
  - `go build ./...`
- Wails 打包（项目根目录）：
  - `wails build`

## 发布目录建议
- 建议将以下文件放在同一发布根目录（Windows）：

```text
MiHoYoStarterGo/
  MiHoYoStarterGo.exe
  config.json
  bin/
    PaddleOCR-json_v1.4.1/
      PaddleOCR-json.exe
      ...（其余 OCR 依赖文件）
```

- `config.json` 建议与主程序 `exe` 同级，避免被外部启动器工作目录影响。
- `bin/PaddleOCR-json_v1.4.1/` 需完整保留，否则 OCR 无法正常工作。

## 调试说明
- 可通过环境变量开启调试日志：`MHY_DEBUG=1`
- 运行中可在状态弹窗使用“保存调试截图”辅助排查 OCR。

## 常见发布问题排查
- 启动后读取不到账号/配置：
  - 检查 `config.json` 是否与 `MiHoYoStarterGo.exe` 同级
  - 若由外部启动器拉起，确认启动器未把配置写到其他目录
- OCR 无结果或报错：
  - 检查 `bin/PaddleOCR-json_v1.4.1/` 是否完整拷贝
  - 重点确认 `PaddleOCR-json.exe`、模型目录、DLL 文件未缺失
- 保存配置失败（如窗口提示权限问题）：
  - 检查发布目录是否有写权限（避免放在受限系统目录）
  - 以管理员方式运行后再次尝试
- 能启动但自动化点击/识别异常：
  - 确认游戏分辨率与标定配置匹配
  - 在状态弹窗保存调试截图，结合 OCR 结果定位问题
